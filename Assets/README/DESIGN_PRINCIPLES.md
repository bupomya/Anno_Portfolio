# Design Principles

> 코드 작성 시 방향과 판단 기준을 정리합니다.
> 새로운 기능을 추가하거나 기존 코드를 수정할 때 이 원칙을 참고합니다.

---

## 1. 코드 작성 우선순위

작업 시 아래 순서로 우선순위를 둡니다:

1. **정확성** - 의도한 대로 동작하는가
2. **단순함** - 지금 필요한 만큼, 가장 간단한 방법인가
3. **가독성** - 다른 사람(미래의 나)이 읽고 이해할 수 있는가
4. **유지보수성** - 수정/확장 시 영향 범위가 명확한가
5. **성능** - 불필요한 연산이 없는가 (단, 과도한 최적화보다 명확함 우선)

> **단순함의 기준은 규모에 따라 변한다.**
> 적 1종류일 때는 코드에서 직접 생성이 단순하지만,
> 적 10종류일 때는 프리팹 + 팩토리가 더 단순하다.
> "지금 이 규모에서 가장 관리하기 쉬운 방법"을 선택한다.

## 2. 추상화 원칙

### 언제 추상화하는가
- **공통 동작이 2개 이상의 구체 클래스에서 반복될 때** 부모 클래스로 추출
- 현재 예시: CharacterBase → PlayerBase → PlayerController / EnemyShooter
  - `CharacterBase`: 모든 캐릭터가 공유하는 것 (생사 상태, 방향, 스프라이트)
  - `PlayerBase`: 모든 플레이어 유형이 공유하는 것 (이동, 점프, 착지, 입력 잠금)
  - `PlayerController`: 특정 플레이어의 고유 동작 (공격, 가드, 패리)

### 추상화 시 생각하는 것
- **"이 동작이 이 클래스의 책임인가?"** → 아니면 분리
- **"자식 클래스가 이것을 바꿔야 할 수 있는가?"** → 그렇다면 virtual/abstract
- **"이 로직을 공유하되, 세부사항만 다른가?"** → Template Method 패턴
  - 현재 예시: `PlayerBase.Update()`가 전체 흐름을 관리하고, `HandleInput()`, `CanMove()`, `OnUpdate()` 등을 자식이 override

### 추상화하지 않는 경우
- 아직 하나의 구현만 존재하고 확장 계획이 불확실할 때
- 추상화가 오히려 코드를 복잡하게 만들 때
- **"나중에 필요하면 그때 추출한다"** (YAGNI 원칙)

## 3. 확장성 설계

### 새로운 캐릭터 추가 시
- `CharacterBase`를 상속하여 새 적 타입 생성 (예: 근접 적, 보스)
- `PlayerBase`를 상속하여 새 플레이어 타입 생성 (예: 궁수, 전사)
- 기존 코드를 수정하지 않고 새 클래스를 추가하는 방향 (OCP)

### 새로운 시스템 추가 시
- 기존 시스템과의 결합도를 최소화
- 싱글톤은 **전역 접근이 반드시 필요한 매니저급**에만 사용
  - 현재: CameraShake, DialogueManager
  - 남용 금지 — 의존성이 보이지 않게 되므로

### 데이터와 로직 분리
- 데이터는 ScriptableObject로 분리 (Inspector에서 편집 가능)
- 현재 예시: DialogueData(데이터) ↔ DialogueManager(로직) ↔ SpeechBubble(표현)
- 새 콘텐츠 추가 시 코드 변경 없이 SO 에셋만 추가

## 4. 유지보수 원칙

### 의존성 방향
- **단방향 의존**을 유지한다
  - Projectile → PlayerController (O) / PlayerController → Projectile (X)
  - DialogueManager → SpeechBubble (O) / SpeechBubble → DialogueManager (X)
- 순환 의존이 생기면 설계를 재검토

### 변경의 영향 범위
- 하나의 변경이 여러 파일에 영향을 주면 인터페이스/추상 클래스 도입 검토
- 현재는 규모가 작으므로 직접 참조를 사용하되, 규모가 커지면 이벤트/인터페이스로 전환

### 컴포넌트 조합 vs 상속
- **상속**: "~이다" 관계 (PlayerController는 PlayerBase이다)
- **컴포넌트 조합**: "~을 가진다" 관계 (캐릭터가 대화 기능을 가진다 → NPCInteraction 컴포넌트 부착)
- 가능하면 컴포넌트 조합을 선호하되, 명확한 계층이 있으면 상속 사용

## 5. 성능 고려사항

### 기본 원칙
- **측정 전에 최적화하지 않는다**
- 단, 아래는 기본으로 지킨다:
  - `Animator.StringToHash()` → static readonly로 캐시
  - `GetComponent<>()` → Awake()에서 한 번만 호출하고 캐시
  - 물리 연산은 `FixedUpdate()`에서 처리
  - `Find` 계열은 `Awake()`/`Start()`에서만 사용

### 오브젝트 생성/파괴
- 현재: Projectile을 매번 생성/파괴 (규모가 작아서 충분)
- 투사체가 빈번해지면 Object Pooling 도입 검토

## 6. 코드 작성 시 체크리스트

새 기능 작성 전:
- [ ] 기존 코드에서 재사용할 수 있는 것이 있는가?
- [ ] 이 기능의 책임은 어느 클래스에 있어야 하는가?
- [ ] 다른 시스템과의 결합을 최소화했는가?

코드 작성 후:
- [ ] 다른 사람이 이 코드를 읽고 의도를 파악할 수 있는가?
- [ ] 이 변경으로 기존 기능이 깨지지 않는가?
- [ ] README 문서 업데이트가 필요한가?

## 7. SOLID 원칙 (참고)

프로젝트 규모에 맞게 단계적으로 적용한다. 과도한 적용은 단순함을 해치므로 균형을 유지한다.

### S - 단일 책임 원칙 (Single Responsibility)
> 클래스는 하나의 책임만 가진다. 변경 이유가 하나뿐이어야 한다.
- 현재 적용: DialogueData(데이터), DialogueManager(흐름), SpeechBubble(표시) 분리
- **항상 지킨다** — 새 클래스 작성 시 "이 클래스의 변경 이유가 2개 이상인가?" 확인

### O - 개방-폐쇄 원칙 (Open/Closed)
> 확장에는 열려있고, 수정에는 닫혀있어야 한다.
- 현재 적용: CharacterBase 상속 구조 → 새 캐릭터는 자식 클래스 추가로 해결
- **항상 지킨다** — 가상 메서드/추상 메서드로 확장 지점 제공

### L - 리스코프 치환 원칙 (Liskov Substitution)
> 자식 클래스는 부모 클래스를 대체할 수 있어야 한다.
- 현재 적용: DialogueManager가 PlayerBase 타입으로 참조 → 어떤 자식이든 동작
- **항상 지킨다** — 자식이 부모의 계약을 깨뜨리지 않도록 주의

### I - 인터페이스 분리 원칙 (Interface Segregation)
> 사용하지 않는 메서드에 의존하지 않아야 한다.
- 현재: 인터페이스 미사용 (규모가 작아서 불필요)
- **도입 시점**: 서로 다른 기능을 가진 캐릭터가 섞일 때 (예: IAttackable, IInteractable, IDamageable)

### D - 의존성 역전 원칙 (Dependency Inversion)
> 상위 모듈이 하위 모듈에 직접 의존하지 말고, 추상화에 의존해야 한다.
- 현재: 직접 참조 (Projectile → PlayerController 등)
- **도입 시점**: 같은 역할을 하는 구현이 여러 개 생길 때 (예: Projectile이 플레이어/NPC 구분 없이 IDamageable만 보면 되는 시점)

### 현재 프로젝트와의 균형
```
항상 적용:  S, O, L (코드 품질의 기본)
규모 커지면: I, D  (지금은 직접 참조가 더 단순)
```
